# PRD · 智能文档排版工具（smart-doc-formatter）

> 本 PRD 由方案文档 + 需求澄清会话综合而成，作为所有实现 issue 的父规格。
> 配套：切片 issue 见 GitHub tracker（label `tracer-bullet`），方案全文见 [docs/proposal.md](./proposal.md)。

## Problem Statement

写文档的人（学生、研究者、投稿作者、机构撰稿人）内容早已写好，却要把它套进某个学校 / 期刊 / 机构规定的**格式标准**——字体字号、页边距、行距、三线表、图表题注、页眉双线、页码分节、参考文献 GB/T 7714…… 手工排版**费时、易漏、易退稿**；而且每家规范不同，今天农大、明天投某期刊，每换一处就得重来一遍。

现有工具靠人海堆模板，**长尾规范**（小众院校、各期刊投稿格式、公司/机构文书）覆盖不到；更关键的是，多数工具在重排过程中**不保证"内容不丢"**——掉一张图、串改一处文字，对一篇要交的论文就是事故。

## Solution

一个工具：用户传一份**「内容已写好」的任意格式文档**（docx / pdf / .doc / …）+ 指定**一份目标格式规范**（从模板库里选，或上传一份全新规范）→ 工具自动：

1. **确定性/ML 提取** 文档为「区块流 + 二进制资产」（文本、表格、图片保留锚点）；
2. **LLM 只做结构分类**（把每段判成标题/正文/图题/三线表/参考文献条目…）；
3. **套用目标格式**：确定性引擎按声明式模板（YAML）查表刷格式；新规范先经「规范编译器」抽成 YAML 并让用户 10 秒确认规则；
4. **双重校验**：格式校验（不合规→自动回炉重排）+ 内容审查（确定性 diff，掉内容即红灯硬阻断）；
5. 产出**「内容一字不丢」的合规 docx + 合规报告**。

核心资产是 **「规范 → 声明式模板（YAML）」编译器 + 可持续增长的模板库**。LLM 只在「结构分类」和「规范抽取」两处登场，其余全是确定性代码——这是「输出即合规、可复现」的根。

## User Stories

1. 作为论文作者，我想把写好内容的 docx 一键套上本校论文格式，这样我不必手工调字体段距页边距。
2. 作为论文作者，我想从模板库里按「省份→学历→院校」选到我学校的规范，这样我不用自己描述格式。
3. 作为论文作者，我想在库里**没有**我学校规范时，直接上传一份学校发的规范文档，工具就能自适应套用，这样冷门院校也能用。
4. 作为论文作者，当我上传一份全新规范时，我想**当场拿到一份标注「未核验·草稿」的成品**先用，同时这份规范进入建设期人工核验队列、核验后自动升级为「已验证」；套用前工具只问我**自己答得上来的问题**（这是硕士还是本科？要英文摘要吗？每章另起页吗？）、主动摊开它**自己没把握的地方**、并**渲染第一页**给我瞄，而不是让我逐项核对页边距/字号这种我根本判断不了的数值。
5. 作为论文作者，我想要排版后**正文一个字都不被改动或丢失**，这样我能信任成品。
6. 作为论文作者，我想要我的**表格被原样保留**（数据不变、结构不乱），只是边框被刷成三线表样式。
7. 作为论文作者，我想要我的**图片不丢、保持在原来的逻辑位置**（跟在原来那段后面），这样图文对应不乱。
8. 作为论文作者，我想要图/表自动套上「图3-1 / 表3.1」的按章编号，并在合规报告里以**醒目、区别化的方式声明「这些编号是工具自动添加的」**，这样既省事、又能一眼核出工具加了什么（编号是机器按结构算出来的格式层产物，可自动加；但工具绝不凭空编造我没写的信息，如文献缺失的卷期页码）。
9. 作为论文作者，我想要中文宋体、数字与西文 Times New Roman 自动分开绑定，这样不出现中英字体混乱。
10. 作为论文作者，我想要摘要/Abstract、关键词/Key words 按规范排（标题居中、关键词加粗、英文摘要另起页），这样前置部分也合规。
11. 作为论文作者，我想要页眉显示论文题目并带粗细双线、页码按「封面无码/前言罗马/正文阿拉伯分节重启」处理，这样这些最烦的细节自动搞定。
12. 作为论文作者，我想要参考文献自动按 GB/T 7714-2015 排、文内引用上标编号，这样不用手工抠文献格式。
13. 作为论文作者，当我有 BibTeX 文献库时，我想直接导入并转成 GB/T 7714，这样文献部分零手工。
14. 作为论文作者，我想要数学公式被转成 Word 原生公式（OMML）并按章编号，这样公式可编辑且合规。
15. 作为论文作者，我想要排版完拿到一份**合规报告**，逐项告诉我每条格式要求过没过，这样我心里有底。
16. 作为论文作者，当某条格式没排对时，我想要工具**自动重排修正**（有上限），而不是把问题甩给我。
17. 作为论文作者，当工具检测到「内容可能丢失」时，我想要它**红灯拦下并明确报告**，而不是悄悄产出残缺成品。
18. 作为投稿作者，我想用同一个工具把同一篇文章套到不同期刊的投稿格式，这样一稿多投不必重排。
19. 作为撰稿人，我想把 pdf 来源的文档也能套格式（中文学术 pdf 的公式表格能被正确解析），这样来源不限 docx。
20. 作为用户，我想扔进**老 .doc、甚至不常见格式**也能被处理（兜底转换），这样我不必先手工另存为 docx。
21. 作为用户，我想要一个简单界面：拖文件进去、选规范、点一下、下载成品 + 报告，这样无需懂命令行。
22. ~~（隐私敏感用户 / 切本地模型）~~ **本条已下线**：v1 明确 all-in 走最强 API 模型求最优效果，不设隐私门槛、不做本地兜底功能；接口层保留切换口子但不作为功能。（见 Out of Scope）
23. 作为工具维护者，我想用规范编译器把一份规范 PDF 自动抽成 YAML 草稿 + 生成 Word 母版，这样新增一份规范不用写代码。
24. 作为工具维护者，我想对编译器产出的 YAML 对一份样张人工校验一次再入库，这样保证该规范后续零误差。
25. 作为工具维护者，我想要每份模板带版本与生效日期、registry 留版本历史，这样规范年度小改时可追溯。
26. 作为工具维护者，我想要一个「规范变更监测 → 一键重编译」流程，这样规范漂移时能快速更新模板。
27. 作为工具维护者，我想要一份 `template.schema.json` 校验每份 YAML 合法（必填项、取值范围），这样模板库不被脏数据污染。
28. 作为开发者，我想把「格式化引擎」包成可被别的程序调用的库/MCP，这样能复用到写作工具、文档工具里。
29. 作为开发者，我想让 LLM 调用封在一个接口后面，这样在 API 与本地模型间切换只改一处。
30. 作为开发者，当遇到 schema 未覆盖的新规范元素时，我想要编译器**主动标记**「需扩展字段」而不是瞎猜，这样异常可见。

## Implementation Decisions

**总体架构**
- **建设期 / 运行期两轨**，中间用**声明式模板库**连接。建设期把规范编译成模板；运行期用模板跑文档。
- **不是 agent，是 workflow**：运行期是一条确定性流水线 + 一个有上限的「校验→不过回炉」循环。**LLM 只在两处登场**：运行期的「结构识别（分类）」、建设期的「规范抽取」。其余全是确定性代码。
- **核心安全不变量 = 每元素「语义载荷守恒」**（取代含糊的「逐字不动」）。审查器按区块类型核对「该守住的信息」有没有被破坏，分三种待遇：
  - 🔴 **用户写入的信息被改/丢 → 红灯硬阻断**（正文串改、文献丢条、表格数据变、图片丢失）。
  - 🟡 **工具按规范重排已有信息 / 自动生成「能算的编号」 → 不红灯，在报告里醒目区别化声明**（文献→GB/T、引用→上标、字体段距、图3-1 / 表3.1 / 公式(1-1) 编号）。
  - ⚪ **得凭空编造的信息（如文献缺失的卷期页码）→ 绝不编造，报告标记让用户补**。
  即：守的是「你写进去的信息一点没少、没被张冠李戴」，而非「排出来的字一模一样」。

**运行期主链**
- **输入路由**：按文件类型**确定性查表**分发到 handler（docx→pandoc/python-docx；pdf→MinerU；.doc→LibreOffice 先转）。**加一种格式 = 加一个 handler**（数据/插件，非 agent）。未知/怪格式兜底走 markitdown / LibreOffice 万能转；仍失败才唤起 **bounded agent 兜底**，工具固定、输出用 Pydantic 区块 schema 卡死、不过重试。**运行期不放 agent 当司机。**
- **提取分格式**：结构化格式（docx/xlsx/pptx）**确定性无损抽**文本/表格/图片二进制；pdf/扫描件才用 **ML（MinerU/版面模型）**还原结构。**图片全程是二进制资产**，贴着逻辑锚点走完全链，LLM 不碰。
- **中间表示**：自定义 **JSON 区块流**，每块带 `type` + 内容。**Pydantic 模型 = 区块契约**，LLM 输出端与排版读取端对同一份 schema。
- **结构识别**：接 **DeepSeek / Qwen API + Instructor**（Pydantic 强约束输出），只做分类不做生成；长文分章节喂。调用封在接口后，保留一行切本地 Ollama+Qwen / Outlines 的口子。
- **目标规范的来源**：运行期由用户提供。库内规范直接取 YAML+母版；**库外新规范走「运行期规范编译器」（混合模式）**：规范文本→LLM 抽规则→draft YAML + **程序化生成** `reference.docx` 母版 → **当场产出标注「未核验·草稿」的成品给用户先用**，**同时**这份规范进入建设期人工核验队列、核验后升级「已验证」供全体复用。**确认环节重做**：不让非专家逐项核对数值，而是只问用户**答得上来的分类/结构问题** + **工具主动摊开自己没把握的项** + **渲染第一页**预览。**同构新规范高可靠；schema 未覆盖的异构元素→主动标记，不瞎猜。注意：三道闸都是拿成品对 YAML，救不了「规范本身被读错」——这正是混合模式靠「诚实摊开不确定性 + 核验队列」兜底的原因。**
- **确定性排版**：**python-docx + 预制 Word 母版**。母版承载页面设置、文档网格、页眉双线、各级命名样式（含中文 eastAsia 字体绑定）。python-docx 填不平的坑用 **oxml/lxml 补丁**：三线表 `tblBorders`（1.5/0.5 磅）、页码 `PAGE` 域 + 分节页码格式、`w:docGrid` 文档网格、页眉段落边框双线。**母版必须能从 YAML 程序化生成**（支撑运行期自适应，不依赖人在 Word 里点）。
- **公式**：pandoc `LaTeX→OMML` 转 Word 原生公式；源公式是图片/MathType 时降级标记。
- **参考文献**：GB/T 7714-2015 顺序编码制 + 文内上标；有 BibTeX 走 `bibtex2gbt7714`，无则 LLM 拆结构化字段后确定性渲染。

**校验与回炉**
- **三道独立校验闸 + 第四张网**：① **格式校验**——重开 docx 逐项断言属性 == 目标 YAML，**驱动自动回炉**。② **内容审查**——**确定性**地按每元素「语义载荷守恒」比对输入/输出（正文文本/表格单元格/图片清单/文献条目与字段），**信息被破坏 = 红灯硬阻断**，绝不靠重排修。③ **结构校验（新增第三道闸）**——**确定性**地拿铁的事实反查 LLM 的分类标签：图题旁必须真有图、表题旁必须真有表、标题编号必须连续、文献条目得长得像文献；**只把真异常抛给用户确认**，堵住「分类错了但格式/内容两闸都绿」的盲区。④ **视觉 QA（第四张网，all-in 加买）**——成品渲染成图，视觉模型挑 property 断言看不出的视觉毛病（错位/压字/孤行），**只进报告、永不改稿**。
- **自动回炉环（B）**：格式不过 → **靶向重刷失败属性** → 再校验，**≤3 次**，超限出报告交人。**回炉只动格式层，绝不碰内容**（内容审查闸始终绿）。区分两类失败：格式属性不对（回炉能修）vs 内容丢失/抽取出错（回炉修不了，直接报人）。

**模板库**
- `templates/<规范>/` 存 `*.yaml` + `_assets/reference.docx`；`registry.json` 总索引（省份→学历→院校→模板ID→版本→状态）；`template.schema.json` 校验每份 YAML 合法。
- 每份模板带 `spec_version` + `updated_at`；支持「提交规范」众包入口（可选）与「变更监测→一键重编译」。

**界面**：自用 Gradio / CLI；服务化 FastAPI；可包成 MCP 供外部驱动。

**LLM 路线 = All-in 求最优效果**：两个 LLM 缝（运行期结构识别、建设期规范抽取）**直接上最强 API 模型 + 喂全文与上下文 + 分类多跑投票取一致**，不做「便宜模型分类、低置信度才升级」的省钱操作，不设隐私/算力门槛。接口层保留切本地的口子但**不作为 v1 功能**。**铁律不变**：钱花在「更多检查网」（投票 + 视觉 QA），不花在「更多驾驶」——LLM 永不当流水线司机、永不碰内容。

## Testing Decisions

**测试哲学**：只测**外部行为**，不测实现细节。即「扔进 X 文档 + Y 规范 → 成品满足这些属性」，不断言内部函数怎么实现。

**测试缝（seam，尽量用最高的）**
- **最高缝 · 端到端**：`(输入文档, 目标 YAML) → (成品 docx, 合规报告)`。**golden / 属性断言测试**：断言成品 docx 的属性（页边距/字体/字号/样式/页眉页码）== YAML，且**内容清单**（段落/表/图/文献条数 + 文本指纹）== 输入。**首选缝**，覆盖整链。
- **引擎缝**（确定性，最该重测）：`(区块流, YAML) → docx`。纯确定性 → 属性级/字节级断言；同输入两次跑结果一致。
- **校验器缝**（三道闸均确定性）：`(docx, YAML) → 格式合规报告`、`(输入, 输出) → 每元素内容守恒报告`、`(区块标签, 资产/表格事实) → 结构异常报告`。确定性，易构造正反样例；**每个元素切片都要带自己那类的「内容守恒」断言**。
- **规范编译器缝**：`规范文本 → YAML(Pydantic)`。对一组规范片段断言抽取字段；含「异构元素应被标记」的负样例。
- **结构识别缝**（含 LLM，非确定）：`区块流 → 标注区块`。测**契约形状**（输出必过 Pydantic 校验）+ 一个小金标集上的分类准确率阈值，**不**断言每条精确标签。
- **内容审查是硬不变量**：每个端到端测试都附带「零内容丢失/零串改」断言。
- **金标校准（农大）**：用一份**人工排到完全合规的农大成品**当「标准答案」——① 从它量出真实属性反推农大 YAML；② 校准格式校验器（已知合规的文档若被报红，是校验器/YAML 错，不是文档错）；③ 把它的格式打乱、内容不动，反向造出「内容写好、没排版」的派生输入，配成 golden 测试对。收口前再补一份真·野生论文作输入。

**Prior art**：全新仓库无既有测试；`siyuanzhou/PaperFormatDetect` 的「提取属性逐项比对」思路可借作格式校验测试的参照。

## Out of Scope

- **降重 / 降 AIGC / 代写 / 查重**——明确不做，划清产品边界。
- **内容生成 / 改写**——工具只换格式，绝不动内容语义。
- **从数据（excel/json）生成文档**（报告生成器 = `数据→模板填充`）——是另一种产品形态，不在本 PRD；若确需，单独立项。本 PRD 的 excel/json 仅指「装着同一份正文的另一种壳」。
- **商业化 / 计费 / 账号 / 分发**——不在本工具范围。
- **物理像素级位置保持**——重排必然改分页，只保**逻辑锚点**，不保「第 X 页坐标」。
- **LaTeX 输出路线**——多数场景只需 Word，不为此入坑。
- **隐私门槛 / 本地模型兜底**——v1 明确 all-in 走最强 API 求最优效果，不做隐私门槛、不 productize 本地模型；接口层留切换口子但不作为功能（原 User Story #22 已下线）。

## Further Notes

- **阶段路线**：阶段0 主链（标题+正文端到端 + **三道闸** + 回炉）+ **农大金标 fixture 就位** + **编译器可行性 spike（最大赌注最早翻牌）** → 元素填充（摘要/三线表/图/文献/公式/页眉页码，每元素带内容守恒断言）→ 输入扩展（pdf + 多格式路由）→ 运行期自适应编译器（15a 程序化母版 / 15b 抽取核心 / 15c 确认UX+混合模式）→ 界面。最硬风险是「运行期规范编译器读错规范无自动兜底」，靠「混合模式：草稿即时 + 核验队列 + 诚实摊开不确定性」化解。
- **隐私（v1 已降级）**：v1 不设隐私门槛（all-in 求最优效果）；接口层保留切本地口子，待真有隐私敏感需求再 productize。
